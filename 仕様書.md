**作成日**: 2026年3月8日
**作成者**: フォーカス株式会社
**バージョン**: 1.0

---

## 1. システム概要

### 目的
スマートフォン・周辺機器のレンタル事業における台帳管理をデジタル化し、以下の課題を解消する。

- どの端末が誰に貸し出されているか一目でわかるようにする
- 契約期限の見落れを防ぐ自動アラートを実現する
- 月次の料金・利益集計を自動化する
- レンタル登録・返却・販売の操作を簡略化する

### 移行元
旧HIS三宮_販売・レンタル用端末管理台帳（Excel）から全データを移行する。

### システム名
**携帯レンタル管理システム**

## 3. 機能一覧

### 優先順位

| 優先 | 機能 | 説明 |
|---|---|---|
| 1 | 契約期限アラート | 終了60日前・30日前・7日前にTeams通知 + 画面表示 |
| 2 | 在庫確認 | どの端末が空いているか一覧で確認 |
| 3 | 料金・利益集計 | 月次集計の画面表示 + Excelダウンロード |
| 4 | レンタル契約登録 | 端末選択→顧客選択→契約内容入力の6ステップ |

### 機能一覧詳細

#### A. 端末管理
- 端末の新規登録・編集（スマートフォン・周辺機器）
- ステータス管理（在庫 / レンタル中 / 販売済み）
- 動作確認チェック記録（外観・起動・SIM・充電・バッテリー）
- 仕入情報記録（仕入先・価格・仕入日・入荷日）

#### B. レンタル契約管理
- レンタル貸出登録（6ステップウィザード形式）
- 返却処理（返却日・状態確認・修理費記録）
- 自動更新フラグ管理
- 最低契約期間・累計契約期間の管理

#### C. 販売管理
- 在庫端末の販売処理（顧客・販売日・販売方法・価格）
- 販売履歴の記録・参照

#### D. アラート
- ダッシュボードに期限アラート一覧を表示（60日・30日・7日以内）
- ダッシュボードから直接「返却処理」「詳細」を実行可能
- Teamsへの自動通知（Logic Apps / 毎日9時実行）

#### E. 料金・利益集計
- 月次レンタル売上・利益の集計（グラフ表示）
- 月次販売売上・利益の集計
- 修理費の利益への反映
- Excelダウンロード（レンタル明細・販売明細シート別）

#### F. お客様連携
- Dynamics 365（Dataverse）の `account` エンティティから顧客を検索・選択
- 契約に顧客IDをリンク（Dataverseがマスタ）

#### G. データ移行
- Excelファイル（.xlsx）のアップロードによる一括インポート
- 「レンタル端末」「レンタル（端末本体以外）」シートの全行を移行
- レンタル中の行は契約情報も同時に移行
- 販売済みの行は販売情報も同時に移行

## 4. 画面一覧と操作フロー

### 画面構成

```

├── ダッシュボード
├── 端末管理
│   ├── 端末一覧
│   └── 端末登録・編集
├── 契約管理

│   └── 返却処理
├── 料金・利益

```

### 各画面の詳細

#### ダッシュボード
| 表示要素 | 内容 |
|---|---|
| サマリーカード | レンタル中台数 / 在庫台数 / 今月売上 / 今月利益 |
| アラート一覧 | 60日・30日・7日以内に終了する契約を色分けで表示 |
| クイックアクション | アラート行から「返却処理」「詳細」をワンクリックで実行 |

#### 端末一覧
| 項目 | 内容 |
|---|---|
| フィルター | ステータス / お客様名 / 契約終了日（期間） / 機種名 / IMEI（この優先順で表示） |
| 表示列 | 管理番号 / 機種名・カラー・容量 / ステータス / お客様名 / 契約開始日 / 契約終了日 / 月額料金 / 仕入価格 / IMEI / 修理履歴有無 |
| 操作ボタン | 在庫→「貸出」「販売」 / レンタル中→「返却」 |

#### レンタル登録（6ステップ）
```
Step1: 端末確認（端末一覧の「貸出」ボタンから起動）
  ↓
Step2: お客様選択（Dataverseから会社名検索）
  ↓
Step3: 契約内容（契約開始日・終了日・期間・自動更新・配送先）
  ↓
Step4: 料金（月額卸価格・月額エンドユーザー価格・利益プレビュー）
  ↓
Step5: 保証（自然故障保険・OP保証・備考）
  ↓
Step6: 確認・登録
```

#### 返却処理
```
① 返却日の入力
② 端末状態の選択（動作OK / 修理が必要）
③ 状態メモの入力
④ 修理が必要な場合 → 修理費・修理内容を入力
⑤ 「返却処理を完了する」ボタン
→ 端末ステータスが「在庫」に戻る
```

#### 販売処理
```
① 端末一覧の「販売」ボタンから起動
② 顧客をDataverseから検索・選択
③ 販売日・販売方法・販売価格を入力
④ 確認・登録
→ 端末ステータスが「販売済み」になる
```

#### 料金・利益画面
| 要素 | 内容 |
|---|---|
| 月・年の選択 | ドロップダウンで対象月を切り替え |
| サマリーカード | レンタル売上 / 販売売上 / 修理費 / 総利益 |
| グラフ | レンタル・販売の売上と利益を棒グラフで表示 |
| レンタル明細テーブル | お客様別・端末別の月額明細（`/api/reports/monthly` の `rentals` フィールドで提供） |
| 販売明細テーブル | 月内の販売実績 |
| Excelダウンロード | レンタル明細・販売明細をシート別でダウンロード |

> **⚠️ 廃止予定に関する注記（2025年末時点）**
>
> Teams の「コネクタ（受信Webhook）」は Microsoft により **廃止予定** となっています。
> 現在の実装（`logicapp-alert.json`）は動作しますが、将来的に動作しなくなる可能性があります。
>
> **推奨移行先: Power Automate ワークフロー**
> - Teams チャンネルに「Power Automate」タブを追加し、「Teams にメッセージを投稿する」アクションを使用
> - Logic Apps → Power Automate への移行で Incoming Webhook 依存を解消できます
> - 移行期限が正式決定した場合は、早めに対応することを推奨します

## 8. アラートルール

| トリガー | 対象 |
|---|---|
| 契約終了の60日前 | status = 'active' かつ contract_end_date = 今日+60日 |
| 契約終了の30日前 | status = 'active' かつ contract_end_date = 今日+30日 |
| 契約終了の7日前 | status = 'active' かつ contract_end_date = 今日+7日 |

- ルールは固定（画面からの変更不可）
- Logic Apps が毎日9時に上記3条件をチェックし、該当件数が1件以上の場合にTeamsへ送信
- ダッシュボードには60日以内の全件をリアルタイム表示

**自動更新契約のアラート挙動**

`auto_renewal = 1`（自動更新あり）の契約も、`contract_end_date` を基準として同一ルールで通知する。
自動更新するかどうかの最終判断はスタッフが行うため、通知をスキップしない。
スタッフはアラートを受け取った後、Teamsまたはアプリ上で返却 / 再契約の手続きを行う。

## 9. APIエンドポイント一覧

| メソッド | パス | 説明 |
|---|---|---|
| GET | /api/devices | 端末一覧（フィルター対応） |
| GET | /api/devices/:id | 端末詳細（修理・販売履歴含む） |
| POST | /api/devices | 端末登録 |
| PUT | /api/devices/:id | 端末更新 |
| POST | /api/devices/:id/sell | 販売処理 |
| GET | /api/contracts | 契約一覧 |
| GET | /api/contracts/:id | 契約詳細（端末情報・修理費合計含む） |
| POST | /api/contracts | レンタル契約登録 |
| POST | /api/contracts/:id/return | 返却処理 |
| GET | /api/alerts | 期限アラート取得（60日以内） |
| GET | /api/customers | Dataverse顧客検索 |
| GET | /api/reports/summary | ダッシュボードサマリー |
| GET | /api/reports/monthly | 月次集計 |
| GET | /api/reports/monthly/export | 月次Excelダウンロード |
| POST | /api/import/excel | Excelデータ移行 |

全エンドポイントはEntra IDトークン認証必須。
`/api/alerts/internal` のみ Functions Key 認証（Logic Apps専用）。

**エンドポイント仕様の補足**

- `PUT /api/devices/:id` — `status` フィールドは変更不可。ステータス変更は専用トランザクション（貸出・返却・販売）を通じてのみ行う。データ修正が必要な場合はAzureポータルで直接DB操作を行うこと。
- `GET /api/devices` の `customer` フィルター — 現在アクティブな契約に紐付く端末のみが対象（`status='active'` の rental_contracts に JOIN）。`in_stock` 端末を顧客名で検索することはできない仕様。
- `GET /api/alerts/internal` — 呼び出し時に `days_until_end` がちょうど 60・30・7 の契約を `alert_logs` に自動記録する。

## 10. デプロイ手順概要

### Step 1 — Entra IDアプリ登録
1. Azureポータル → アプリの登録 → 新規作成
2. 名前: `rental-management`
3. リダイレクトURI（SPA）: `http://localhost:3001`（開発用）、Static Web AppsのURL（本番用）
4. APIの公開 → スコープ `access_as_user` を追加
5. APIのアクセス許可 → Dynamics 365 API（`user_impersonation`）を追加
6. テナントID・クライアントIDを控える

### Step 1.5 — ローカル開発環境の準備（開発者向け）

ローカル開発時は Azure Storage Emulator（Azurite）が必要です。

```bash
npm install -g azurite
azurite --silent &   # バックグラウンドで起動
```

`local.settings.json` の `AzureWebJobsStorage: "UseDevelopmentStorage=true"` がAzuriteを参照します。Azuriteが起動していないと Functions の起動に失敗します。

### Step 2 — Azureリソース作成
```bash
cd infra
export ENTRA_CLIENT_ID="<クライアントID>"
export ENTRA_TENANT_ID="<テナントID>"
bash deploy.sh
```

### Step 3 — DBスキーマ適用
Azure SQL Database に接続し `database/schema.sql` を実行。

### Step 4 — フロントエンド環境変数設定
`frontend/.env.local` を作成:
```
VITE_ENTRA_CLIENT_ID=<クライアントID>
VITE_ENTRA_TENANT_ID=<テナントID>
VITE_DATAVERSE_URL=https://org5a73169f.crm7.dynamics.com
```

### Step 5 — Teamsチャンネル・Webhook設定
1. Teams に「レンタル管理アラート」チャンネルを作成
2. チャンネルの「コネクタ」→「受信Webhook」→ URLを発行
3. Logic Apps（`infra/logicapp-alert.json`）にWebhook URLを設定してデプロイ

### Step 6 — Excelデータ移行
1. アプリにログイン
2. 設定画面 → Excelデータ移行
3. 旧台帳ファイルをアップロードしてインポート実行
